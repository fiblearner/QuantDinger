
## 二、标的池筛选

**文件：** `scripts/fetch_top_symbols.py`

### 筛选流程

1. 连接 Binance USDM 永续合约，拉取全量 24h ticker
2. 过滤条件：
   - 仅保留 `BASE/USDT` 或 `BASE/USDT:USDT` 格式
   - 24h 成交额 ≥ 500 万 USDT
   - Base symbol 长度 2~10 个字符，纯字母数字
   - 排除稳定币（USDC/BUSD/DAI 等）、指数（BTCDOM/DEFI）、杠杆代币（3L/3S/BULL/BEAR）
   - 过滤价格 < 0.000001 USDT 的垃圾币
   - 过滤 24h 涨跌幅绝对值 > 50% 的暴涨暴跌币（新上线 meme）
3. 按成交额降序，取前 200 个作为候选（`TOP_N × 2`）
4. **逐一验证 K线可用性**：拉取 4H OHLCV 5 根，能返回 ≥ 3 根才入选
5. 最终保留前 100 个有效标的

### 输出格式

```json
{
  "symbol_list": ["Crypto:BTC/USDT", "Crypto:ETH/USDT", ...],
  "detail": [{"symbol": "BTC/USDT", "volume_usdt": 2800000000, "change_24h": 1.23}, ...]
}
```

> **注意：** `symbol_list` 中带 `"Crypto:"` 前缀，执行引擎内部会自动归一化为 `"BTC/USDT"` 后与数据库 raw symbol 对齐。

## 三、K 线处理流程

**数据源：** Binance USDM，4H K线，最近 **400 根**
**约覆盖：** 约 66 天历史数据

### 步骤 1：K 线包含处理

按缠论规则合并包含关系，消除伪分型噪音：

```
向上趋势中遇到包含：取两者 max(high), max(low)
向下趋势中遇到包含：取两者 min(high), min(low)
```

处理后得到无包含关系的 `(ph, pl)` 序列。

### 步骤 2：顶底分型识别

三根 K 线条件：
- **顶分型**：中间根的 high > 左右两根，且 low > 左右两根
- **底分型**：中间根的 low < 左右两根，且 high < 左右两根

### 步骤 3：笔的合并

- 顶底必须交替出现
- 两个分型之间间距 ≥ 4 根 K 线（包含处理后）
- 同向出现时取极值（顶取更高、底取更低）

### 步骤 4：中枢识别

三笔及以上有重叠区间即构成中枢：
- **ZG（中枢顶）** = 三笔中最低的高点
- **ZD（中枢底）** = 三笔中最高的低点
- `ZG > ZD` 才是有效中枢
- 相邻中枢重叠时自动合并扩展

## 四、买点评分逻辑

> 当前策略**仅做多**（`long_ratio=1.0`），卖点评分不参与信号生成。

### 辅助指标

| 指标 | 参数 | 用途 |
|------|------|------|
| MACD | EMA12 / EMA26 / Signal9 | 面积背驰判断（一类买点） |
| RSI | 14 周期，EMA 平滑 | 买点加权修正 |
| 成交量 | 原始 volume | 量能背驰确认（一类买点） |
| MA | 收盘价 240 根均线（≈60 日） | 趋势过滤：价格须在均线以上 |

### 入场前置过滤（两项，任一不满足则 score = 0）

1. **趋势过滤**：收盘价 ≥ 60 日均线（4H × 240 根）
2. **量能萎缩过滤**：近 14 根均量 / 近 90 根均量 < 70% 时视为量缩，需当日成交量 ≥ 近 90 根均量 × 2.0，或近 4 根均量 ≥ 前 40 根均量 × 2.5 才放行

---

### 4.1 二类买点（满分 75）

**条件：**
1. 存在有效中枢
2. 中枢结束后最后一笔方向向上
3. 该笔起点（底）> 中枢底 ZD（不跌入中枢）
4. 当前价格距该底 ≤ 12%（`dist < 0.12`）

**评分公式：**
```
s2 = 75 × (1 - dist / 0.12)
```

**止损位：** 中枢底 ZD × 0.98（留 2% 缓冲）

---

### 4.2 三类买点（满分 65）

**条件：**
1. 存在有效中枢（至少 5 笔）
2. 中枢后有一笔向上突破（终结笔顶 > ZG）
3. 随后一笔向下回调，回调底 > ZG（不回入中枢）
4. 当前价格距回调底 ≤ 8%（`dist < 0.08`）

**评分公式：**
```
s3 = 65 × (1 - dist / 0.08)
```

**止损位：** 前中枢顶 ZG × 0.98（留 2% 缓冲）

---

### 4.3 RSI 加权修正

| 条件 | 调整 |
|------|------|
| RSI < 35（超卖） | `score × 1.25`（上限 100） |
| RSI > 70（超买） | `score × 0.6` |

---

### 4.4 信号阈值过滤

| 买点类型 | 入选阈值 | 满分 |
|---------|---------|------|
| 二类买点 | **≥ 60** | 75（RSI 加权后可超 80） |
| 三类买点 | **≥ 60** | 65（RSI 加权后可超 80） |

`score < 60` 或 `stop_loss_price <= 0` 或 `stop_loss_price ≥ 当前价格` 时信号归零，不参与排名。

## 五、截面选股逻辑

每日调仓时：
1. 对 100 个标的全部计算评分，过滤后只保留 `score > 0` 的有效信号
2. 按评分从高到低排序（`rankings`）
3. 取前 5 名作为目标多仓列表（`long_symbols`）
4. 对 **不在目标列表中但已持仓** 的标的：**不主动平仓**，等待止损或量缩横盘条件触发

> **设计原则：** 持仓入场有门槛，出场有独立逻辑。避免频繁换仓导致手续费侵蚀，与回测逻辑保持一致。
>
> **满仓 5 个时：** 出现新高评分标的不会触发换仓，等待现有持仓因止损/量缩退出后，下次调仓才补入新标的。

## 六、资金与杠杆管理

| 参数 | 值 | 说明 |
|------|-----|------|
| `initial_capital` | **750 USDT** | 初始资本（对应账户实际余额） |
| `leverage` | 2× | 合约杠杆 |
| `entry_pct` | 20% | 每仓使用当前净值比例 |
| `portfolio_size` | 5 | 最多同时持仓数 |
| `long_ratio` | 1.0 | 仅做多，不做空 |

**资金分配计算（动态净值口径）：**

```
available_capital = initial_capital + 历史已实现盈亏 - 当前未平仓亏损
单仓保证金 = available_capital × 20%
单仓名义价值 = 单仓保证金 × 2（杠杆）
开仓数量 = 单仓名义价值 ÷ 当前价格
```

示例（净值 750 USDT，BTC 价格 103,000 USDT）：
```
单仓保证金 = 750 × 20% = 150 USDT
单仓名义 = 150 × 2 = 300 USDT
开仓张数 = 300 ÷ 103,000 ≈ 0.00291 BTC
```

## 七、止损与风控

### 7.0 盘中实时止损（每 5 分钟）

每个 tick 都会执行，**不依赖调仓时间**。

1. 读取所有持仓的 `stop_loss_price`、`entry_price`、`trail_level`
2. 拉 Binance REST 实时价格
3. 计算浮盈，更新 trail_level（只升不降）
4. 价格穿透有效止损价时立即发 `close_long` 信号

去重保护：同一标的触发后 300 秒内不重复发单；DB 中已有 `pending/processing` 的平仓单时跳过。

---

### 7.1 结构止损（入场时设定）

| 买点类型 | 止损位 |
|---------|--------|
| 二类买点 | 中枢底 ZD × **0.98** |
| 三类买点 | 前中枢顶 ZG × **0.98** |

止损触发条件：`当前价格 < effective_stop`，**立即平仓，不受最短持仓时间约束**。

**止损方向校验：** 开仓前检查 `stop_loss_price < 当前价格`，若止损位 ≥ 当前价格则拒绝开仓（信号结构无效）。

---

### 7.2 阶梯锁利止损（trail_level 机制）

浮盈达到阈值时，止损逐级上移（只升不降，DB 持久化）：

| trail_level | 触发浮盈 | 有效止损 | 含义 |
|-------------|---------|---------|------|
| 0 | 初始 | 结构止损位 | — |
| 1 | ≥ **10%** | max(结构止损, 成本价 × 1.0) | 保本 |
| 2 | ≥ **50%** | max(结构止损, 成本价 × 1.2) | 锁利 20% |
| 3 | ≥ **100%** | max(结构止损, 成本价 × 1.5) | 锁利 50% |
| 4 | ≥ **200%** | max(结构止损, 成本价 × 2.0) | 锁利 100% |
| 5 | ≥ **300%** | max(结构止损, 成本价 × 3.0) | 锁利 200% |

```python
def _trail_stop_mult(level: int) -> float:
    if level == 1: return 1.0   # 保本
    if level == 2: return 1.2   # 锁利20%
    if level == 3: return 1.5   # 锁利50%
    return float(level - 2)     # level4→×2.0, level5→×3.0 …
```

---

### 7.3 量缩横盘退出

持仓一段时间后，若价格未有效突破且成交量萎缩，触发平仓：

| 参数 | 值 | 含义 |
|------|-----|------|
| `VOL_CONSOL_BARS` | 42 根 | 观察窗口（4H × 42 ≈ 7 天） |
| `VOL_CONSOL_PCT` | +8% | 价格突破阈值 |
| `VOL_SHRINK_RATIO` | 0.60 | 量能萎缩阈值 |
| `VOL_MIN_HOLD_H` | **72h** | 最短持仓时间（进入量缩检测前） |

**触发条件（同时满足）：**
```
1. 持仓时长 ≥ 72h
2. 近 42 根 K 线最高收盘价 ≤ entry_price × 1.08（价格未突破 8%）
3. 近 42 根平均成交量 < breakout_vol（入场时记录）× 0.60
```

触发后立即平仓，记录为 `close_long`。

---

### 7.4 冷却期（止损/量缩后）

止损或量缩横盘平仓后，**同一标的 5 天内禁止重新开仓**。

```sql
SELECT COUNT(*) FROM qd_strategy_trades
WHERE strategy_id = ? AND symbol = ?
  AND type IN ('close_long', 'close_short')
  AND created_at >= NOW() - INTERVAL '5 days'
```

冷却期内该标的即使出现新信号也跳过。

---

### 7.5 最短持仓时间

`VOL_MIN_HOLD_H = 72`（3 天）

- 量缩横盘检测：持仓 < 72h 的标的跳过（避免误判）
- 结构止损不受此限制，可立即平仓

## 八、调仓执行流程

```
每 5 分钟 tick
  ├── [1] 盘中实时止损检查（每 tick 必执行，见 7.0）
  ├── [2] 标的列表刷新检查（symbol_refresh_days=5，每 5 天刷新一次，只更新列表）
  └── [3] 调仓时间判断：当前北京时间是否在 00:00~00:10 窗口内且今天尚未调仓
           └── 满足条件 → 完整调仓流程（约需 3~8 分钟）
```

**调仓时间：** 每天北京时间 **00:05**（`rebalance_time="00:05"`，±5 分钟窗口，每天只触发一次）

### 标的列表自动刷新

| 参数 | 值 | 说明 |
|------|-----|------|
| `symbol_refresh_days` | 5 | 每 5 天自动从 Binance 拉取最新成交量排行更新标的池 |
| `last_symbol_refresh_at` | DB 持久化 | 重启后继续计时，不重置 |

刷新只更新 `symbol_list`，**不触发调仓**，等下次 00:05 调仓时才使用新列表。

### 完整调仓步骤

```
执行截面指标（串行拉取 100 标的 × 400 根 4H K线 + 评分）
        ↓
生成信号列表（按优先级）：
  ① 止损信号（跌破 effective_stop，立即平仓，不受持仓时间限制）
  ② 量缩横盘退出信号（持仓 ≥ 72h，价未突破且量缩）
  ③ open_long（新进入 Top5 且未持仓，通过冷却期/止损方向校验）
  （不在 Top5 的现有持仓不主动平仓）
        ↓
开仓日志记录：
  [开仓信号] BTC/USDT open_long | 评分=78.5 类型=二买 止损=95000 当前=102000
        ↓
并行执行所有信号（最多 10 线程）
        ↓
补写 breakout_vol（记录入场时突破量，用于后续量缩检测）
        ↓
记录调仓时间（last_rebalance_at）
```

## 十二、已知局限与后续优化方向

| 问题 | 现状 | 可能方向 |
|------|------|---------|
| K 线串行拉取慢 | 100 标的逐一请求，首次调仓约 3~8 分钟 | 并发拉取或引入 K 线缓存服务 |
| 中枢识别精度 | 仅用 3 笔最小中枢，大级别中枢可能被拆分 | 增加线段层级识别 |
| 满仓不换仓 | 5 仓满后新信号忽略，直到有仓位退出 | 加入评分差阈值换仓逻辑 |
| 卖点未开空 | `long_ratio=1.0`，卖点信号无实际执行 | 开启空仓池（`long_ratio<1`） |
| 调仓频率固定每日 | 错过日内结构变化 | 支持信号变化触发的随时调仓 |
| 一类买点未启用 | 当前 INDICATOR_CODE 只识别二买/三买 | 加入 MACD 面积背驰检测恢复一买 |
